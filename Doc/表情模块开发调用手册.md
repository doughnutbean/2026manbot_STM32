# 表情模块开发调用手册

更新时间：2026-05-28

## 1. 目标与范围

本文档用于统一项目中“表情模块”的调用方式，覆盖：

- 表情指令与表情 ID 的对应关系
- `MainTask` 到 `DisplayTask` 的消息流
- 上游模块（语音/业务）推荐调用方法
- 下游模块（显示驱动）消费约定
- 后续扩展新表情时的修改规范

当前实现基于 CMSIS-RTOS2 消息队列，表情由 `MB_DisplayEvent_t` 承载。

## 2. 相关代码位置

- 协议与事件定义：`App/Inc/mb_protocol.h`
- 表情映射与分发：`App/Src/mb_tasks.c`
- 队列创建：`App/Src/mb_app.c`

## 3. 核心数据结构

### 3.1 指令枚举 `MB_CommandId_t`

表情相关指令（协议层统一）：

- `MB_CMD_EXPRESSION_HAPPY = 0x30`
- `MB_CMD_EXPRESSION_SAD = 0x31`
- `MB_CMD_EXPRESSION_SLEEPY = 0x32`
- `MB_CMD_EXPRESSION_CONFUSED = 0x33`

### 3.2 表情 ID 枚举 `MB_ExpressionId_t`

- `MB_EXPRESSION_NONE = 0x00`
- `MB_EXPRESSION_HAPPY = 0x01`
- `MB_EXPRESSION_SAD = 0x02`
- `MB_EXPRESSION_SLEEPY = 0x03`
- `MB_EXPRESSION_CONFUSED = 0x04`

### 3.3 显示事件 `MB_DisplayEvent_t`

```c
typedef struct
{
		MB_CommandId_t command;
		uint8_t expression_id;
		uint8_t animate;
} MB_DisplayEvent_t;
```

字段说明：

- `command`：触发表情的源命令，便于日志与联调追踪。
- `expression_id`：具体表情资源 ID（建议只使用 `MB_ExpressionId_t` 定义值）。
- `animate`：是否播放动画，`1` 为动画，`0` 为静态。

## 4. 命令到表情映射（当前实现）

由 `MB_MapCommandToDisplay()` 统一管理，映射如下：

| 源命令 | expression_id | animate | 说明 |
|---|---:|---:|---|
| `MB_CMD_FORWARD` | `MB_EXPRESSION_HAPPY` | 1 | 运动联动表情 |
| `MB_CMD_BACKWARD` | `MB_EXPRESSION_HAPPY` | 1 | 运动联动表情 |
| `MB_CMD_WAVE` | `MB_EXPRESSION_HAPPY` | 1 | 运动联动表情 |
| `MB_CMD_SIT` | `MB_EXPRESSION_SLEEPY` | 0 | 静态表情 |
| `MB_CMD_TURN_LEFT` | `MB_EXPRESSION_CONFUSED` | 1 | 转向联动表情 |
| `MB_CMD_TURN_RIGHT` | `MB_EXPRESSION_CONFUSED` | 1 | 转向联动表情 |
| `MB_CMD_EXPRESSION_HAPPY` | `MB_EXPRESSION_HAPPY` | 1 | 直接表情命令 |
| `MB_CMD_EXPRESSION_SAD` | `MB_EXPRESSION_SAD` | 0 | 直接表情命令 |
| `MB_CMD_EXPRESSION_SLEEPY` | `MB_EXPRESSION_SLEEPY` | 0 | 直接表情命令 |
| `MB_CMD_EXPRESSION_CONFUSED` | `MB_EXPRESSION_CONFUSED` | 1 | 直接表情命令 |

说明：

- 既支持“运动触发表情联动”，也支持“独立表情指令直达显示模块”。
- 对于表情直达命令，当前不触发 TTS（`MB_TTS_NONE`）。

## 5. 调用链路

### 5.1 上游生产者（例如语音模块）

上游仅需发送 `MB_VoiceEvent_t.command`，无需直接写显示队列：

```c
MB_VoiceEvent_t evt = {
		.command = MB_CMD_EXPRESSION_HAPPY,
		.timestamp_ms = 0U,
		.confidence = 100U
};
(void)osMessageQueuePut(g_mb_app.voice_to_main_queue, &evt, 0U, 0U);
```

`MainTask` 会自动完成：

1. 指令解析
2. 表情映射（必要时同时派发动作/TTS）
3. 将 `MB_DisplayEvent_t` 推送至 `main_to_display_queue`

### 5.2 下游消费者（显示模块）

`DisplayTask` 从 `main_to_display_queue` 取消息后，建议按如下规范实现：

1. 校验 `expression_id` 是否在合法范围（`MB_EXPRESSION_HAPPY` 到 `MB_EXPRESSION_CONFUSED`）。
2. 根据 `expression_id` 选择资源帧组。
3. 根据 `animate` 决定播放静态帧还是动画序列。
4. 记录 `command -> expression_id` 日志，便于联调回放。

## 6. 队列与并发约束

- 队列：`main_to_display_queue`（长度默认 `MB_QUEUE_LEN_DISPLAY = 8`）。
- 设计原则：
	- 上游不跨层直写显示队列，统一经 `MainTask` 仲裁。
	- 表情映射只在一处维护（`MB_MapCommandToDisplay()`），避免多处散落。
	- `DisplayTask` 尽量不阻塞过长时间，避免积压后延迟放大。

## 7. 扩展新表情规范

新增一个表情（例如“惊讶”）建议按顺序操作：

1. 在 `MB_CommandId_t` 增加 `MB_CMD_EXPRESSION_SURPRISED`。
2. 在 `MB_ExpressionId_t` 增加 `MB_EXPRESSION_SURPRISED`。
3. 在 `MB_MapCommandToDisplay()` 增加对应分支，并明确 `animate` 策略。
4. 在显示驱动资源表中加入新表情帧组。
5. 补充联调日志与测试用例（直达表情命令、与动作联动命令）。

## 8. 常见问题

### 8.1 为什么不建议业务层直接发 `MB_DisplayEvent_t`？

因为会绕开主控仲裁，导致动作、TTS、状态机与显示不同步。统一走 `MB_VoiceEvent_t -> MainTask` 更易维护。

### 8.2 为什么有些表情 `animate = 0`？

这是当前策略选择：`SAD`、`SLEEPY` 默认静态，`HAPPY`、`CONFUSED` 默认动画。可在 `MB_MapCommandToDisplay()` 中集中调整。

### 8.3 显示效果不对先查哪里？

先查 `MB_MapCommandToDisplay()` 输出，再查显示模块资源索引映射，最后看队列是否积压。

## 9. 联调最小检查单

- 发送 `MB_CMD_EXPRESSION_HAPPY` 后，`DisplayTask` 是否收到 `MB_EXPRESSION_HAPPY`。
- 发送 `MB_CMD_EXPRESSION_SLEEPY` 后，`animate` 是否为 `0`。
- 发送 `MB_CMD_TURN_LEFT` 后，是否同时触发动作与 `MB_EXPRESSION_CONFUSED`。
- 高频连续发送表情命令时，显示任务是否出现明显积压或掉帧。
