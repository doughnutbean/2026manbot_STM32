## 4. 当前架构符合度检查

结合现有源码，当前工程整体符合 `AGENTS.md` 中描述的阶段一基本架构。

### 4.1 启动流程

从 `Core/Src/main.c` 可见，当前启动流程为：

1. `SystemClock_Config()`
2. `BoardLed_Init()`
3. `DebugUart_Init()`

5. `MB_AppInit()`
6. `MB_AppStartTasks()`

这与项目约定的“系统时钟 -> 板载 LED -> 调试串口 -> RTOS -> App 初始化 -> 启动任务”基本一致。

- `voice_to_main_queue`
- `main_to_motion_queue`
- `main_to_display_queue`
- `main_to_tts_queue`

已启动以下任务：

- `MB_TaskMain`
- `MB_TaskVoice`
- `MB_TaskMotion`
- `MB_TaskDisplay`
- `MB_TaskTts`
- `MB_TaskHeartbeat`

这说明当前项目已经形成“语音输入 -> 主任务路由 -> 动作/显示/TTS 执行”的基本消息转发框架，符合阶段一任务边界设计。

### 4.3 状态管理

### 6.2 协议头文件存在注释编码异常

在 `App/Inc/mb_protocol.h` 中，可以看到部分中文注释已经出现乱码现象，例如新增命令与新增 TTS 的注释行。

影响：

- 降低可维护性
- 容易在后续扩展命令枚举时造成误读
- 可能说明部分文件编码未统一

建议后续统一检查源码文件编码，优先采用项目当前可稳定兼容的编码格式，并修复乱码注释。

### 6.3 `mb_tasks.c` 存在源码直接包含 `.c` 文件的现象

当前 `App/Src/mb_tasks.c` 中直接包含了：

- `mb_face_bitmap.c`
- `mb_servo.c`
- `mb_motion.c`

注释说明是为了避免 Keil 单独编译。

该做法短期可运行，但有以下风险：

- 增加编译单元耦合
- 容易引发重复定义或构建配置混乱
- 不利于后续模块独立调试与复用

建议后续结合 Keil 工程分组与编译选项，逐步恢复为正常的头文件声明 + 独立源文件编译方式。

### 6.4 测试注入宏当前处于开启状态

在 `App/Src/mb_tasks.c` 中：

- `MB_DEBUG_LOG_ENABLE` 为 `1U`
- `MB_TEST_INJECT_ENABLE` 为 `1U`

说明当前更偏向调试联调版本，而非纯运行版本。  
当 `MB_TEST_INJECT_ENABLE == 1U` 时，语音任务会进入动作测试流程，而不是正常语音监听流程。

这对当前阶段调试是合理的，但若后续需要进行真实语音链路验证，需要明确关闭该宏并记录切换行为。
