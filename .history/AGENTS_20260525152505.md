# AGENTS Guide for 2026manbot_STM32

This file helps coding agents become productive quickly in this repository.

## Project Context

- MCU: STM32F103VE (Cortex-M3)
- Toolchain/IDE: Keil MDK5, ARMCC5 (legacy ARM-ADS toolset)
- RTOS: CMSIS-RTOS2 with RTX5
- Entry point: [Core/Src/main.c](Core/Src/main.c)
- App framework: [App/Src/mb_app.c](App/Src/mb_app.c), [App/Src/mb_tasks.c](App/Src/mb_tasks.c)

## Build and Validation Workflow

- Primary build workflow is Keil GUI using [manbot.uvprojx](manbot.uvprojx) and target "Target 1".
- Expected successful result: 0 errors and 0 warnings.
- Build log reference: [Objects/manbot.build_log.htm](Objects/manbot.build_log.htm).
- Runtime smoke check on hardware: heartbeat LED on PB0 toggles about every 500 ms.

If CLI automation is unavailable in the current environment, do not invent alternative build systems.

## Architecture and Boundaries

- Startup flow: System clock -> board LED init -> debug UART init -> RTOS init -> app init -> start tasks.
- Task pipeline: Voice -> Main -> Motion/Display/TTS via message queues.
- App context and IPC objects are centralized in [App/Inc/mb_app.h](App/Inc/mb_app.h).
- Protocol enums, events, and shared contracts are defined in [App/Inc/mb_protocol.h](App/Inc/mb_protocol.h).

When changing behavior, keep these boundaries stable:
- Prefer extending command/event structs and handlers over bypassing queues.
- Keep state changes mediated by the app state lock.
- Avoid direct cross-task coupling that skips Main task routing.

## Repository-Specific Conventions

- Prefer editing files under App and Core for feature work.
- Do not modify vendor/framework code under Drivers unless explicitly requested.
- Do not edit generated outputs under Objects and Listings for source changes.
- Debug and test inject defaults are controlled in [App/Src/mb_tasks.c](App/Src/mb_tasks.c):
  - MB_DEBUG_LOG_ENABLE
  - MB_TEST_INJECT_ENABLE

## Agent Working Checklist

Before coding:
- Read startup flow and protocol definitions.
- Confirm whether changes impact pin usage, queue topology, or command enums.

After coding:
- Keep API and task topology consistent unless change is explicitly requested.
- Verify there is no accidental edit to Drivers, Objects, or Listings.
- If hardware behavior changed, note expected runtime observables for validation.
